# 資産・契約台帳メーカー

自分の**資産・契約台帳**（銀行・カード・サブスク・保険・年金・不動産・アカウントなどの所在一覧）を
作って**印刷する**ための、**ログイン不要・サーバー非保持の静的ブラウザツール**。
台帳の下書き作成、カード明細からの定期課金（サブスク）抽出、CSV/PDF の書き出しまで、
すべてブラウザ内で完結します。終活・相続の資産整理や、サブスクの解約準備に使えます。

> 設計原則: このアプリは**足場 (scaffolding)** であって構造体ではありません。
> 明日このアプリが落ちても、ユーザーの備え（印刷した紙＋物理保管）は一切壊れません。
> 詳細は [`BUILD_INSTRUCTION.md`](./BUILD_INSTRUCTION.md) を参照。

## スコープ

**「個人が自分の資産・契約台帳を作り、印刷する」**ことだけに絞っています。
相続手続きの代行、資産運用や法務・税務の助言、家族間の共有・通知などは行いません。
台帳の外で知っておくとよいこと（認知症への備え、相続手続き、サブスク解約など）は
`/guide` に参考情報としてまとめています。

## 保管モデル（地図と鍵を分ける）

- 台帳は「**どこに何があるか**」を示す**地図**です。**パスワード・口座番号・PIN 等の秘匿情報は台帳に保存しません**（入力欄を設けていません）。
- 台帳は **印刷して物理保管**（家庭用金庫・鍵付き引き出し＋封筒・銀行の貸金庫）します。アプリの保存は揮発しうる下書きで、**正本は印刷した紙**です。
- 鍵にあたる秘匿情報は、**別途まとめて同じ保管先**に置きます（台帳とは分離）。
- いざという時の引き継ぎは、**保管場所と開け方を平時に家族へ伝えておく**ことで担います。台帳には秘匿情報が含まれないため、地図としてのデジタル共有も可能です。

## ハードルール（実装が守っていること）

- バックエンドにユーザーデータを保存しない（サーバーDB・永続ストレージなし）
- 認証機能を持たない（ログイン・セッション・トークンなし）
- 外部APIにユーザーデータを送らない（LLM不使用。明細解析もすべてブラウザ内）
- 秘匿情報（パスワード等）は台帳のスキーマに持たない
- 正本はアプリの外（印刷した紙 ＋ 物理保管）。IndexedDB は揮発しうる下書き

## 技術スタック

- Next.js（Pages Router）+ TypeScript / 静的エクスポート（`output: 'export'`）
- Tailwind CSS
- IndexedDB（`idb`、下書き専用。UIの折りたたみ状態のみ localStorage）
- PDF生成: `pdf-lib` + `@pdf-lib/fontkit`（Noto Sans JP を `public/fonts/` に同梱して埋め込み）
- CSVパース: `papaparse` / CSV生成は自前（依存なし、BOM付きでExcel対応）

## 開発

```bash
npm install
npm run dev      # http://localhost:3000
npm run build    # 静的エクスポート → out/
```

`npm run build` で `out/` に静的サイトが生成されます。Vercel 等にそのままデプロイできます
（サーバー実行経路・API Routes は持ちません）。

## データモデル（要点）

- 台帳 = レコードの集合。各レコードは型（銀行口座／クレカ／サブスク／保険／年金／不動産／端末・アカウント／連絡先／自由メモ）を持つ。
- **公開/非公開の区分は廃止**。台帳は地図情報のみを持ち、書き出しは1種類（PDF=印刷・閲覧用 / CSV=再編集用の正本）。
- 各フィールドは **必須/任意** と **PCレイアウトでの相対幅** を持つ。必須項目は赤い `*` で表示し、必須が埋まればそのカテゴリは「入力済み」になる（任意項目は空欄でも可）。必須は「死亡時にスムーズに対応できる」観点で、口座・カード・契約を特定し止める/請求するのに要る項目を選定。
- カテゴリは全件が最初から表示される。各カテゴリは複数アイテム（例: 銀行口座を複数）を持て、「該当なし」で無効化、見出しの ± ボタンで最小化できる（最小化状態は localStorage に保存）。
- 再編集の正本は**CSV**。`/export` の「CSVを読み戻す」で読み込み、編集を再開できる（旧フォーマットの visibility 列は無視して読み込む）。

## 明細取込（サブスク抽出）

- 複数ファイルをまとめて取り込み可能（1ヶ月ずつの明細などを分けて選んでも加算）。文字コードは UTF-8 / Shift_JIS を自動判定。
- **列（日付・金額・摘要）とヘッダー有無はデータ内容から自動推定**（日付として読める率・金額らしさ・テキスト率・充足率・ばらつき・長さで判定）。プリセット・手動調整も可能。
- 自動で定期課金を絞り込むのではなく、**正規化キーで店舗・サービス別にまとめる**だけ。周期・金額一致は並び順とおすすめチェックのヒントに留め、**どれをサブスクとして取り込むかはユーザーが目視で選択**する。完全に重複する明細（同一日付・店名・金額）は自動で集約。
- 取り込み前に「紐づくカード・口座」を入力でき、取り込む各サブスクの該当フィールドに反映される。

## フォントについて

PDF の日本語表示には Noto Sans JP（OFL）を `public/fonts/NotoSansJP-Regular.ttf` に同梱し、
`pdf-lib` でサブセット埋め込みしています。標準フォントは日本語を出力できません。
